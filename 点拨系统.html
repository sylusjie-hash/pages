这是一份将“行为诊断结果”转成“教学干预动作”的结构化需求稿。页面围绕背景与目标、用户与场景、功能与流程、约束与边界四大模块展开，适合用作方案对齐、汇报展示和产品落地参考。

1 诊断到干预的闭环目标

系统定位

点拨系统不是重新判错的模块，而是基于已确认错误模式，生成提醒、资源和下一步动作建议的后置决策层。

读取诊断快照 匹配点拨策略 检索资源库 输出结构化点拨
  • 先诊断，再点拨，避免泛化讲题。
  • 只消费结构化数据，不直接回查原题或 Excel。
  • 为 APP 前端提供可展示、可追踪、可承接的输出结果。

01 / Background & Goals

背景与目标

明确为什么要做点拨系统，以及 V1 需要真正解决的业务问题。

背景

当前系统已经具备行为诊断和错误模式识别能力，能够判断学生在解题过程中的某一步动作是否完成，并在动作未完成时进一步识别错误模式。

但诊断结果仍停留在“识别问题”层，尚未形成针对错误的标准化点拨输出，系统还没有完成“诊断后干预”的真正闭环。

建设目标

  1. 在错误模式已确认的前提下，自动生成与当前错误相匹配的点拨结果。
  2. 将点拨结果标准化输出给 APP 或上层业务系统。
  3. 通过“对话点拨 + 资源推荐 + 下一步动作建议”形成干预闭环。
  4. 让点拨逻辑可配置、可维护、可扩展，而不是依赖临时人工编写。

业务目标

提高纠错效率 减少无效讲解 沉淀标准映射 支撑统一接口

一句话概括

将“诊断结果”转化为“可执行的教学干预动作”，让系统从会发现问题，升级到会推动纠偏。

02 / Users & Scenarios

用户与场景

明确点拨系统服务谁、出现在哪些关键节点，以及它在业务流里的作用方式。

学生 / 考生

在做题过程中出现动作错误或思路偏差时，接收针对性点拨、卡片、课程或练习建议。

教研 / 内容运营

维护点拨策略库和资源库，配置不同错误模式下的点拨目标、方式和资源内容。

APP / 业务系统

调用点拨接口，获取结构化结果，并完成前端展示、推荐和后续动作承接。

核心使用场景

  1. 学生在某一步解题动作中被判定为未完成。
  2. 系统进一步识别并确认学生当前属于某个错误模式。
  3. 诊断结果写入运行时快照后，点拨系统被触发。
  4. 系统根据错误模式、动作类型和题型，匹配最合适的点拨策略。
  5. 输出点拨话术、资源卡片、专项练习和下一步建议。
  6. APP 展示结果，学生据此重试、补学或专项训练。
场景 A：过早看选项 学生没有完成“识别题干关键判断对象”，系统确认其错误模式为“过早看选项”，此时应提示其先回到题干圈出判断对象，并推荐对应点拨卡片与验证练习。
场景 B：理解偏差但未完全错误 学生表现为理解偏差但未完全错误时，系统应优先返回轻量对话点拨，而不是直接推送整段课程内容。
场景 C：无可用策略或资源 当错误模式明确，但当前没有可用策略或资源时，系统应返回“无匹配策略”或“暂不点拨”状态，避免输出不准确内容。
03 / Features & Flow

功能与流程

定义点拨系统要做哪些事，以及它如何从诊断快照走到结构化点拨输出。

功能 1：点拨触发判断

根据诊断快照中的 ready_for_hint 判断当前是否进入点拨环节。

功能 2：点拨策略匹配

根据 error_mode_id + action_id + question_type 组合匹配点拨策略。

功能 3：点拨资源检索

根据策略中的 resource_refs 获取课程、卡片、练习或文章等资源。

功能 4：点拨结果组装

将策略内容与资源内容组装为统一输出，包括点拨类型、提醒内容、资源、验证计划、下一步建议和追踪信息。

点拨结果输出字段

decision_type summary_blocks dialogue_hints resource_blocks verification_plan next_step trace

支持的点拨输出状态 / 类型

dialogue card course exercise mixed not_ready no_policy_found

主流程

1

诊断与确认

上游系统先完成动作判定；若动作未完成，则进入错误模式识别，并将结果写入最新诊断快照。

2

触发点拨

点拨系统读取快照；若 ready_for_hint=false，返回“暂不点拨”；若为 true，则继续执行匹配。

3

匹配策略与资源

按错误模式、动作和题型匹配点拨策略，再根据资源引用列表获取对应资源内容。

4

组装并返回结果

系统输出结构化点拨结果，前端据此引导学生进行重试、补学、进入下一步或专项验证。

策略匹配规则

  1. 优先按 confirmed_error_mode_id 精确匹配。
  2. 必须同时匹配 action_id 与 question_type。
  3. 命中多条策略时，按 priority 选择主策略。
  4. 仅允许使用已发布状态的策略与资源。
  5. 若引用资源缺失，应返回受控异常状态，而不是输出不完整点拨。

输出承接规则

  1. dialogue 类型优先在 dialogue_hints 中输出轻量提醒。
  2. card、video 或 material 类型应进入 resource_blocks。
  3. 需要后续验证时，补充 verification_plan。
  4. next_step 必须明确告知后续动作，如重试、进入下一步或专项训练。
04 / Constraints & Boundaries

约束与边界

明确点拨系统该做什么、不该做什么，以及 V1 的范围控制。

系统边界

  1. 不负责判断“当前动作是否完成”。
  2. 不负责重新推断学生属于哪种错误模式。
  3. 不是通用讲题系统，不直接基于原题生成完整解析。
  4. 只负责在诊断结果明确后，生成对应点拨决策输出。

数据边界

  1. 只消费诊断结论快照、点拨策略库和点拨资源库三类结构化数据。
  2. 不直接回查原始题库、Excel 或非结构化教研资料。
  3. 不读取未发布状态的策略或资源。
  4. 缺少 confirmed_error_mode_id 时，允许返回“暂不点拨”。

流程约束

  1. 只有在 ready_for_hint=true 时才能正式触发点拨。
  2. 点拨必须发生在行为诊断和错误模式确认之后。
  3. 点拨结果必须是结构化输出，不能仅输出自由文本。
  4. 结果中需保留 trace 信息，便于回溯命中的动作、错误模式和策略。

V1 范围控制

重点解决：

  • 标准化点拨决策链路
  • 基于错误模式的规则化匹配
  • 对话点拨、卡片、课程、练习等基础输出
  • 支持 APP 侧直接调用与展示

暂不覆盖：

  • 开放式个性化长文本讲解
  • 自动生成全新资源内容
  • 跨题或跨知识点长期路径规划
  • 未确认错误模式下的推断式点拨
点拨结果必须与当前错误模式强相关，语言要简洁、明确、可执行，资源推荐要与点拨目标一致，并且整体结果应具备可复盘、可追踪、可持续维护的属性。
05 / Conclusion

需求结论

用一句产品定义收束整份需求文档。

结论

点拨系统 V1 的本质，是一个位于“诊断完成之后”的结构化决策模块。它不重新判断学生错在哪里，而是在错误已经明确的前提下，快速、稳定地为学生生成“该怎么被提醒、该看什么资源、接下来该做什么”的点拨结果，并把这一结果以标准接口输出给业务系统，形成完整的诊断闭环。
